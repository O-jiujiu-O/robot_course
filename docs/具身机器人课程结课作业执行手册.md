# 具身机器人基础与仿真实践：结课作业执行手册

> 适用环境：Ubuntu 20.04 LTS、NVIDIA 独立显卡、已有 Anaconda 和 CUDA 环境  
> 文档目的：从零完成课程要求，并为最终“详细的过程和技术文档.docx”及汇报 PPT 留下可复现记录  
> 核心原则：不修改已有 Conda 环境，不升级或重装 CUDA/显卡驱动，不在 Conda base 中安装包  
> 技术路线：ROS Noetic + Gazebo Classic 11 + 自制移动机器人/机械臂联合 URDF + Isaac Gym Preview 4 + OpenCV + Vosk

## 0. 先读本节：项目范围与风险

### 0.1 最终要做出的成果

本作业最终实现一个简化的“移动操作机器人”：

- 下部是差速移动底盘，可以前进、后退、转向和停止。
- 上部是两自由度机械臂，可以抬起、放下和挥手。
- 前部有仿真相机，可以识别场景中的红色方块。
- 控制层封装统一 API，例如 `move_forward()`、`turn_left()`、`arm_wave()`、`stop()`。
- 任务编排层可以执行连续行为，例如：
  1. 等待“开始任务”语音指令；
  2. 相机检测红色方块；
  3. 若检测到红色方块，机器人向前移动；
  4. 停车；
  5. 机械臂挥手；
  6. 返回初始姿态。
- 同一个机器人 URDF 模型导入 Isaac Gym，设置质量、摩擦系数、关节驱动模式等物理参数，并运行简单关节运动。

这套成果能够覆盖课程要求，但不包括以下高难度内容：

- 不做真实 SLAM、自主导航、路径规划。
- 不做机械臂逆运动学和真实抓取。
- 不训练强化学习策略。
- 不接入大语言模型。
- 不部署到真实机器人，只说明仿真到真机的迁移方法。

这些限制是合理的。课程要求是“简单任务”“掌握基础方法”，不是在一个结课作业里重新实现工业机器人系统。

### 0.2 为什么选择这套版本

| 组件 | 固定版本/方案 | 原因 |
|---|---|---|
| 操作系统 | Ubuntu 20.04 LTS | 用户现有环境；ROS Noetic 的主要支持平台 |
| ROS | ROS Noetic Desktop Full | 与 Ubuntu 20.04 匹配，包含常用机器人开发工具 |
| Gazebo | Gazebo Classic 11 | ROS Noetic 常见配套版本 |
| 模型格式 | URDF/Xacro | ROS、Gazebo 和 Isaac Gym 均可使用 |
| Isaac Gym | Preview 4 | NVIDIA 发布的 Isaac Gym 最后公开预览版 |
| Isaac Gym Python | 独立 Python 3.8 Conda 路径环境 | 避免污染已有环境，并提高旧软件兼容性 |
| 视觉 | OpenCV HSV 红色检测 | 逻辑直观、结果稳定、容易演示 |
| 语音 | Vosk 离线识别 | 不依赖云 API，适合演示 |

### 0.3 必须知道的兼容风险

1. **Gazebo Classic 11 已于 2025 年 1 月 29 日结束生命周期。**  
   它仍适合完成 ROS Noetic 教学作业，但不适合作为新工业项目的长期技术路线。

2. **ROS Noetic 已于 2025 年 5 月 31 日结束生命周期。**  
   ROS 官方说明其已有二进制包不会立刻消失，但不再提供新功能、安全更新和常规错误修复。本项目选择它，是因为 Ubuntu 20.04、ROS Noetic 与 Gazebo Classic 11 构成成熟且资料丰富的课程复现组合，而不是因为它仍是新项目推荐方案。

3. **Isaac Gym Preview 4 是旧版、闭源预览软件，NVIDIA 已将其标记为 deprecated。**  
   它需要从 NVIDIA Developer 的归档页面下载，不能通过普通 `apt` 安装。IsaacGymEnvs 仓库也已于 2026 年 4 月 14 日归档。对非常新的 GPU、驱动或系统组合，可能出现兼容问题。

4. **课程要求指定 Isaac Gym，所以主路线仍使用 Isaac Gym。**  
   若 Isaac Gym 因新显卡或驱动无法运行，应保留完整错误截图，并在报告中增加 Isaac Lab/Isaac Sim 作为后继方案的说明，但不能假装 Isaac Gym 已成功运行。

5. **本手册禁止执行以下命令：**

   ```bash
   sudo apt upgrade
   sudo apt full-upgrade
   sudo apt install nvidia-driver-*
   sudo apt install cuda
   sudo apt install nvidia-cuda-toolkit
   pip install <任何包>        # 在未确认环境时禁止
   conda install <任何包>      # 在 base 或已有环境中禁止
   ```

6. `sudo apt install` ROS/Gazebo 软件会修改 Ubuntu 系统软件包，但不会修改已有 Conda 环境。执行前仍应阅读 apt 给出的安装和删除列表；如果它准备删除 CUDA、NVIDIA 驱动或大量已有包，立即输入 `n` 取消。

## 1. 课程要求的外行版解释

### 1.1 什么是具身机器人

具身机器人是具有物理或仿真“身体”，能够通过传感器感知环境、进行决策，并用执行器作用于环境的智能系统。

可以将它理解为一个闭环：

```text
环境 -> 传感器 -> 感知 -> 决策/任务规划 -> 控制 -> 执行器 -> 环境变化
```

本项目中各部分的对应关系如下：

| 抽象概念 | 本项目中的实现 |
|---|---|
| 身体 | 移动底盘、机械臂、相机 |
| 环境 | Gazebo 中的地面、墙、红色方块 |
| 感知 | 相机图像与语音输入 |
| 决策 | 任务编排程序 |
| 控制 | 底盘速度命令、机械臂关节命令 |
| 行动 | 移动、停车、挥手机械臂 |
| 反馈 | 相机画面、关节状态、机器人位姿 |

### 1.2 具身智能与传统机器人的区别

| 对比项 | 传统机器人 | 具身智能机器人 |
|---|---|---|
| 典型工作环境 | 固定、结构化，例如工厂产线 | 开放、变化、不完全可预测 |
| 任务方式 | 预先编程、重复执行 | 根据感知和指令动态决策 |
| 感知能力 | 可有可无，通常较单一 | 通常融合视觉、语言、触觉等 |
| 控制逻辑 | 固定流程、确定性强 | 感知与决策闭环，具有适应性 |
| 泛化能力 | 换任务通常需重新编程 | 目标是通过学习或通用模型适应新任务 |
| 例子 | 固定焊接机械臂 | 可听懂指令并在房间中完成任务的移动操作机器人 |

二者不是完全对立。具身智能系统底层仍然依赖传统机器人的运动学、动力学、控制和安全机制。所谓“智能”不能替代稳定的底层控制。

### 1.3 行业现状

具身机器人行业目前主要集中在以下方向：

- **人形机器人：** 强调通用身体形态、双足运动和双臂操作。
- **移动操作机器人：** 移动底盘加机械臂，技术成熟度通常高于完整人形机器人。
- **仓储与物流：** 搬运、分拣、上下料。
- **制造业：** 柔性装配、检测、协作操作。
- **家庭与服务：** 清洁、递送、辅助服务，环境复杂度较高。
- **仿真与合成数据：** 使用 Gazebo、Isaac Sim、MuJoCo 等平台降低真机试验成本。
- **视觉语言动作模型：** 将视觉、自然语言和机器人动作统一建模，但可靠性、安全性和数据成本仍是难点。

行业当前的核心问题包括：

- 真机数据昂贵，采集速度慢。
- 仿真与现实之间存在差异，即“Sim-to-Real Gap”。
- 长序列任务容易累积误差。
- 真实环境中的安全性和可靠性要求高。
- 通用能力展示很多，但稳定、低成本、可量产的产品较少。

报告中不要写“具身智能已经成熟并即将全面替代人类”。这不是事实。更准确的结论是：研究和投资活跃，局部任务已经实用，但通用具身智能仍处于快速发展和验证阶段。

### 1.4 Gazebo、ROS 和 Isaac Gym 分别做什么

**ROS** 不是操作系统内核，而是一套机器人软件通信框架和工具集。不同程序通过 Topic、Service、Action 等机制通信。

**Gazebo** 是机器人仿真器，擅长可视化场景、传感器模拟，并与 ROS 联动。它适合演示机器人在场景中的行为。

**Isaac Gym** 是 NVIDIA 的 GPU 并行物理仿真工具，主要面向大量并行环境和强化学习。它适合展示 GPU 仿真、物理参数配置和批量训练思想。

本项目不强行让 Gazebo 与 Isaac Gym 同时运行。正确做法是让它们复用同一个 URDF 模型，分别完成不同实验。

### 2.1 计划使用的 ROS 接口

| 类型 | 名称 | 数据类型 | 用途 |
|---|---|---|---|
| Topic | `/cmd_vel` | `geometry_msgs/Twist` | 控制底盘速度 |
| Topic | `/joint1_position_controller/command` | `std_msgs/Float64` | 控制机械臂关节 1 |
| Topic | `/joint2_position_controller/command` | `std_msgs/Float64` | 控制机械臂关节 2 |
| Topic | `/camera/image_raw` | `sensor_msgs/Image` | Gazebo 相机图像 |
| Topic | `/vision/red_target` | `std_msgs/Bool` | 是否检测到红色目标 |
| Topic | `/voice/command` | `std_msgs/String` | 识别出的语音命令 |
| Topic | `/task/status` | `std_msgs/String` | 当前任务状态 |

## 4. 阶段二：安装 ROS Noetic 与 Gazebo 11

### 4.1 检查是否已经安装

```bash
which roscore
rosversion -d
gazebo --version
```

理想输出：

```text
noetic
Gazebo multi-robot simulator, version 11.x.x
```

若已经满足要求，跳到 4.4。不要重复安装。

### 4.2 安装 ROS Noetic

以下命令使用 Ubuntu 系统包，不进入 Conda 环境。先确保终端提示符前没有 `(base)` 或其他环境名：

```bash
conda deactivate
```

如果执行一次后仍显示 `(base)`，再次执行，直到退出 Conda 环境。

截至 2026 年，ROS Noetic 已结束支持，但 ROS 官方说明 Focal 二进制包仍会保留在 `packages.ros.org`。因此以下命令用于课程旧环境复现，不代表 ROS 官方仍在维护这些包。若软件源未来不可用，应停止并记录错误，不要随意添加来历不明的软件源。

确认 ROS 终端使用系统 Python：

```bash
which python3
```

应输出 `/usr/bin/python3`。如果输出 Anaconda 路径，说明仍未退出 Conda 环境。

配置 Ubuntu 软件源组件：

```bash
sudo apt update
sudo apt install curl gnupg2 lsb-release software-properties-common
sudo add-apt-repository universe
```

添加 ROS 软件源和密钥：

```bash
sudo sh -c 'echo "deb http://packages.ros.org/ros/ubuntu $(lsb_release -sc) main" > /etc/apt/sources.list.d/ros1-latest.list'
curl -s https://raw.githubusercontent.com/ros/rosdistro/master/ros.asc | sudo apt-key add -
sudo apt update
```

安装：

```bash
sudo apt install ros-noetic-desktop-full
sudo apt install python3-rosdep python3-rosinstall python3-rosinstall-generator python3-wstool build-essential
sudo apt install ros-noetic-gazebo-ros-pkgs ros-noetic-gazebo-ros-control
sudo apt install ros-noetic-ros-control ros-noetic-ros-controllers
sudo apt install ros-noetic-xacro ros-noetic-joint-state-publisher-gui
sudo apt install ros-noetic-cv-bridge python3-opencv
```

在每次 `apt install` 确认前，阅读将被删除的软件包列表。若出现要删除 NVIDIA/CUDA 或已有关键软件，输入 `n` 取消并记录截图。

### 4.3 初始化 rosdep

```bash
sudo rosdep init
rosdep update
```

如果提示 `/etc/ros/rosdep/sources.list.d/20-default.list already exists`，说明以前初始化过，只执行：

```bash
rosdep update
```

### 4.4 配置 ROS 环境

向 `~/.bashrc` 追加 ROS Noetic 环境配置前先检查：

```bash
grep -n "ros/noetic/setup.bash" ~/.bashrc
```

若没有输出，再执行：

```bash
echo 'source /opt/ros/noetic/setup.bash' >> ~/.bashrc
source ~/.bashrc
```

不要把 Conda base 自动激活和 ROS 环境混在一起运行 Isaac Gym。ROS 终端与 Isaac Gym 终端应分开使用。

### 4.5 验证 ROS 与 Gazebo

终端 A：

```bash

```

终端 B：

```bash

```

## 6. 阶段四：理解并验证联合机器人模型

### 6.1 URDF/Xacro 是什么

URDF 是 ROS 中描述机器人结构的 XML 格式，主要包含：

- `link`：刚体，例如底盘、车轮、机械臂连杆。
- `joint`：刚体之间的连接，例如旋转关节、固定关节。
- `visual`：外观。
- `collision`：碰撞几何。
- `inertial`：质量和惯量。
- Gazebo 插件：差速驱动、相机、ROS 控制等。

Xacro 是带变量和宏的 URDF 模板语言，可减少重复代码。运行时会展开成标准 URDF。

### 6.2 本项目模型组成

模型至少包含：

```text
base_footprint
└── base_link
    ├── left_wheel_link
    ├── right_wheel_link
    ├── caster_link
    ├── arm_base_link
    │   └── arm_link_1
    │       └── arm_link_2
    └── camera_link
        └── camera_optical_link
```

关键设计：

- 两个主动轮通过 `continuous` 关节连接到底盘。
- 万向轮可使用球体简化，减少控制复杂度。
- 机械臂关节使用 `revolute` 类型，并设置角度上下限。
- 相机固定在底盘前上方。
- 每个动态 link 必须有合理质量和惯量，否则 Gazebo 可能抖动或飞散。

### 6.3 检查 Xacro 与 URDF

```bash
cd ~/robot_course/catkin_ws
rosrun xacro xacro src/embodied_robot_demo/urdf/embodied_robot.urdf.xacro > /tmp/embodied_robot.urdf
check_urdf /tmp/embodied_robot.urdf
```

正确结果应列出机器人 link 和 joint 树，不应出现 XML 解析错误。

保存模型结构证据：

```bash
check_urdf /tmp/embodied_robot.urdf | tee ~/robot_course/evidence/final_integration/02_gazebo/urdf-check.txt
```

### 6.4 使用 RViz 检查模型

```bash
roslaunch embodied_robot_demo display.launch
```

若代码包没有单独的 `display.launch`，可直接进入 Gazebo 验证，不必为展示而增加不必要文件。

检查：

- 车轮位置是否与底盘匹配。
- 机械臂是否位于底盘上方。
- 关节转动方向是否合理。
- 相机是否朝向机器人前方。

### 6.5 常见模型错误

| 现象 | 常见原因 | 处理方法 |
|---|---|---|
| 模型生成后飞走 | 惯量错误、碰撞体重叠、初始位置穿入地面 | 检查惯量和碰撞几何，抬高出生位置 |
| 轮子转但车不动 | 摩擦不足、轮轴方向错误、插件 joint 名称错误 | 检查 wheel joint、摩擦参数与轴方向 |
| 机械臂下垂或剧烈抖动 | PID 不合适、质量过大、控制器未启动 | 检查控制器和 PID |
| 相机画面方向错误 | optical frame 旋转关系错误 | 修正固定关节姿态 |
| `check_urdf` 报错 | XML/Xacro 语法错误 | 按报错行号修改 |

### 6.6 本阶段验收

- `check_urdf` 成功。
- 能说明 link、joint、visual、collision、inertial 的区别。
- 机器人模型结构完整，无明显穿模和飞散。

## 8. 阶段六：封装机器人控制 API

### 8.1 为什么要封装 API

直接在任务代码中反复构造 ROS 消息会导致逻辑混乱。控制 API 将底层 Topic 发布封装为易懂的方法，使任务层只关心“做什么”，而不是“消息怎么发”。

计划提供的方法：

```python
move_forward(speed, duration)
move_backward(speed, duration)
turn_left(speed, duration)
turn_right(speed, duration)
stop()
set_arm(joint1, joint2, wait_time)
arm_home()
arm_wave(cycles)
```

### 8.2 API 的安全边界

- 线速度限制在 `[-0.4, 0.4] m/s`。
- 角速度限制在 `[-0.8, 0.8] rad/s`。
- 机械臂命令限制在 URDF 定义的关节范围内。
- 每次定时移动结束后必须调用 `stop()`。
- 节点退出时必须发送停车命令。

这不是多余设计。机器人控制程序如果异常退出后仍保持上一条速度命令，真机可能继续运动。

### 8.3 运行 API 演示

Gazebo 运行后，新终端执行：

```bash

source ~/robot_course/catkin_ws/devel/setup.bash
rosrun embodied_robot_demo robot_control_api.py
```

API 演示应顺序执行：

1. 前进。
2. 停止。
3. 左转。
4. 停止。
5. 机械臂挥手。
6. 机械臂回到初始位置。

## 9. 阶段七：视觉交互

### 9.1 视觉功能原理

视觉节点订阅 `/camera/image_raw`，使用 `cv_bridge` 将 ROS 图像转成 OpenCV 图像，再执行：

1. BGR 图像转换为 HSV。
2. 对红色设置两个 HSV 阈值区间。
3. 合并二值掩膜。
4. 使用形态学开闭运算减少噪声。
5. 查找面积最大的红色轮廓。
6. 面积超过阈值时发布 `/vision/red_target = true`。
7. 在调试画面中画出检测框。

红色在 HSV 色相轴两端，因此通常要使用两个区间，而不是一个区间。

### 9.2 启动视觉节点

保持 Gazebo 运行，新终端执行：

```bash

source ~/robot_course/catkin_ws/devel/setup.bash
rosrun embodied_robot_demo vision_node.py
```

另一个终端观察结果：

```bash
rostopic echo /vision/red_target
```

预期：

- 相机看到红色方块时输出 `data: True`。
- 将方块移出视野或转动机器人后输出 `data: False`。

### 9.3 视觉节点验收

保存：

- 带检测框的视觉窗口截图。
- `/vision/red_target` 输出日志。

```bash

```

### 9.4 应理解的局限

HSV 颜色识别不是通用视觉智能。它容易受光照、材质和颜色相近物体干扰。它在本作业中的作用是清楚展示“图像感知结果驱动机器人行为”的闭环。

## 11. 阶段九：多模态任务编排

### 11.1 连贯任务定义

最终演示任务固定为：

```text
初始状态：机器人静止，机械臂在 home 位，相机朝向红色方块。

收到“开始任务”：
1. 发布状态：WAITING_FOR_VISION。
2. 等待视觉节点确认红色目标，超时 10 秒。
3. 若未检测到目标，发布状态：TARGET_NOT_FOUND，并结束。
4. 若检测到目标，发布状态：TARGET_FOUND。
5. 机器人以 0.15 m/s 前进 2 秒。
6. 停车。
7. 机械臂挥手两次。
8. 机械臂返回 home 位。
9. 发布状态：TASK_COMPLETED。

任意时刻收到“停止”：
1. 立即停车。
2. 中止后续步骤。
3. 发布状态：TASK_ABORTED。
```

### 11.2 为什么这算多模态联动

- 语言模态决定“何时开始”和“何时停止”。
- 视觉模态决定“是否允许执行目标相关动作”。
- 任务编排将两种感知结果转换为底盘和机械臂动作。
- 动作执行后改变仿真环境中的机器人状态，形成感知到行动的闭环。

只同时启动相机和麦克风不叫联动。必须让两类输入真正影响行为逻辑。

| GPU/Pipeline 报错 | 驱动、GPU 架构或 Isaac Gym 旧二进制不兼容 | 保存完整日志，尝试 CPU pipeline 或 headless；不要擅自升级驱动 |
| 窗口无法打开 | 远程桌面、无显示环境、图形依赖问题 | 先测试 `--headless` |
| PyTorch/CUDA 报错 | 版本组合不兼容 | 以 Isaac Gym 自带安装说明为准，在专用环境内调整 |
| `libpython` 或动态库错误 | 环境路径或 Python 版本不匹配 | 确认使用 Python 3.8 专用环境 |

如果必须调整 Python、PyTorch 或其他包，只能在 `~/robot_course/envs/isaacgym` 内进行，并在实验记录中写明变更。

## 13. 阶段十一：在 Isaac Gym 导入机器人并配置物理参数

### 13.1 准备标准 URDF

Isaac Gym 通常导入标准 URDF，不直接处理 ROS launch，也不需要 Gazebo 插件和 ROS transmission 标签。完整项目代码包已经提供经过整理的标准 URDF：

```text
~/robot_course/isaacgym_demo/assets/embodied_robot.urdf
```

先直接检查该文件：

```bash
source /opt/ros/noetic/setup.bash
check_urdf ~/robot_course/isaacgym_demo/assets/embodied_robot.urdf
```

Gazebo 使用的 Xacro 与 Isaac Gym 使用的标准 URDF描述同一个机器人，但 Isaac 版本移除了 Gazebo 专属插件、相机传感器插件和 ROS 控制 transmission。不要直接用 Xacro 展开结果覆盖该文件；旧版 Isaac Gym 对额外标签的兼容性有限。

### 13.2 导入脚本需要完成的内容

`import_robot.py` 应实现：

1. 创建 Isaac Gym 仿真。
2. 选择 PhysX 物理引擎。
3. 设置时间步长，例如 `dt = 1/60`。
4. 设置重力，例如 `(0, 0, -9.81)`。
5. 添加地面。
6. 从 `assets/embodied_robot.urdf` 导入机器人。
7. 配置 URDF 导入选项。
8. 创建一个或多个环境。
9. 设置刚体形状摩擦系数。
10. 设置机械臂关节驱动模式、刚度和阻尼。
11. 周期性改变机械臂目标位置。
12. 输出机器人自由度、刚体数量和参数。

### 13.3 建议的物理参数

| 参数 | 建议值 | 含义 |
|---|---:|---|
| `dt` | `1/60 s` | 仿真时间步长 |
| `substeps` | `2` | 每个时间步的物理子步数 |
| 重力 | `-9.81 m/s²` | 地球重力 |
| 地面静摩擦 | `1.0` | 静止时抗滑动能力 |
| 地面动摩擦 | `0.8` | 滑动时摩擦 |
| 关节 stiffness | `200` | 位置控制刚度初始值 |
| 关节 damping | `20` | 位置控制阻尼初始值 |
| arm 目标角 | 在关节限位内 | 禁止越界 |

这些数值是教学演示起点，不是经过系统辨识得到的真机参数。

### 13.4 运行自制模型导入脚本

```bash
conda activate ~/robot_course/envs/isaacgym
cd ~/robot_course/isaacgym_demo
python import_robot.py
```

保存日志：

```bash
python import_robot.py 2>&1 | tee ~/robot_course/evidence/final_integration/03_isaacgym/import-robot.txt
```

### 13.5 物理参数对比实验

至少完成一个可解释的对比实验。推荐做摩擦系数对比：

| 实验组 | 摩擦系数 | 观察内容 |
|---|---:|---|
| A | 0.1 | 接触时更容易滑动 |
| B | 1.0 | 接触更稳定 |

或做关节参数对比：

| 实验组 | stiffness | damping | 观察内容 |
|---|---:|---:|---|
| A | 50 | 2 | 响应慢或振荡明显 |
| B | 200 | 20 | 响应更稳定 |

每组保存截图、参数和观察结论。不要只写“效果更好”，应说明是否振荡、是否超调、是否滑动。

## 14. 阶段十二：仿真到真机部署基础方法

本作业不要求实际真机，但报告必须说明迁移方法。

### 14.1 保持上层接口不变

仿真和真机应尽量使用相同上层接口：

```text
任务编排 -> RobotControlAPI -> /cmd_vel、机械臂控制接口
```

替换到真机时，主要替换的是底层驱动：

| 仿真 | 真机 |
|---|---|
| Gazebo 差速驱动插件 | 电机驱动器和底盘控制器 |
| Gazebo 相机插件 | USB/深度相机驱动 |
| 仿真关节控制器 | 机械臂伺服驱动 |
| 仿真里程计 | 编码器、IMU、定位系统 |

### 14.2 Sim-to-Real Gap 的来源

- 仿真摩擦、质量、惯量与真机不一致。
- 传感器存在噪声、延迟和丢帧。
- 电机存在死区、回差和饱和。
- 真实光照和背景变化更复杂。
- 网络与计算延迟影响控制。

### 14.3 基础迁移流程

1. 对真机建立准确 URDF。
2. 标定相机内参和外参。
3. 测量质量、惯量和关节限制。
4. 使用真机驱动替换 Gazebo 插件。
5. 保持 Topic/API 尽量一致。
6. 先架空轮子或低速测试。
7. 增加急停、限速、碰撞保护和通信超时。
8. 逐步从单动作测试到完整任务。
9. 使用真机数据修正仿真参数。
10. 对感知算法加入噪声和场景变化测试。

### 14.4 真机安全底线

- 真机必须有物理急停。
- 首次测试必须限速。
- 机械臂工作空间内不得站人。
- 通信中断时默认停车。
- 软件停止命令不能替代物理急停。
- 未验证的策略不得直接全速上真机。
