# 具身机器人课程结课作业：三人并行分工执行手册

> 本文档用于将结课作业拆分给三个人并行完成。  
> 原始技术基准仍为《具身机器人课程结课作业执行手册.md》。  
> 本文档只负责分工、执行顺序、交付物和集成方法；遇到技术参数冲突时，以原始执行手册为准。  
> 适用环境：Ubuntu 20.04、ROS Noetic、Gazebo Classic 11、Isaac Gym Preview 4。
> 默认协作场景：A、B、C 分别使用三台互不共享文件系统的 Linux 机器，最终在一台指定集成机上合并和验收。

---

## 0. 三人最终要共同完成什么

最终成果是一个移动机器人与两自由度机械臂组成的联合仿真系统：

- Gazebo 中包含移动底盘、机械臂、相机、简单房间和红色目标。
- 底盘能够前进、后退和转向。
- 机械臂能够执行关节运动和挥手动作。
- 相机能够识别红色目标。
- Vosk 能够识别“开始任务”“停止”等语音命令。
- 任务编排程序能够将语音、视觉、底盘和机械臂动作连接起来。
- 同一机器人模型能够导入 Isaac Gym，并完成物理参数配置和对比实验。
- 最终提交详细过程和技术文档 `.docx`、汇报 PPT、源码、日志、截图和视频。

### 0.1 固定分工

| 人员 | 主要职责 | 负责的课程要求 |
|---|---|---|
| A：Gazebo 与理论负责人 | 理论基础、ROS/Gazebo、联合模型、场景、底盘与机械臂基础验证 | 要求 1、要求 2 |
| B：控制与多模态负责人 | 控制 API、视觉、语音、任务编排、多模态联动 | 要求 4、要求 5 中的交互部分 |
| C：Isaac Gym 与材料负责人 | Isaac Gym、物理参数实验、Sim-to-Real、最终文档和 PPT 汇总 | 要求 3、要求 5 中的部署方法 |

### 0.2 为什么这样分

三人的主要工作对象不同：

- A 主要处理 XML、Xacro、Gazebo world、ROS 控制器和理论内容。
- B 主要处理 Python ROS 节点与多模态逻辑。
- C 主要处理 Isaac Gym Python、物理参数实验和最终材料。

这样可以避免三个人反复修改同一文件。A 和 B 的唯一主要交接点是 ROS Topic；A 和 C 的唯一主要交接点是标准 URDF；B 和 C 的主要交接点是实验结果与最终文档材料。

### 0.3 不允许擅自改变的公共接口

三人不得自行重命名以下 Topic：

| Topic | 数据类型 | 生产者 | 消费者 |
|---|---|---|---|
| `/cmd_vel` | `geometry_msgs/Twist` | B 的控制 API | A 的 Gazebo 底盘插件 |
| `/joint1_position_controller/command` | `std_msgs/Float64` | B 的控制 API | A 的机械臂控制器 |
| `/joint2_position_controller/command` | `std_msgs/Float64` | B 的控制 API | A 的机械臂控制器 |
| `/camera/image_raw` | `sensor_msgs/Image` | A 的 Gazebo 相机 | B 的视觉节点 |
| `/vision/red_target` | `std_msgs/Bool` | B 的视觉节点 | B 的任务编排节点 |
| `/voice/command` | `std_msgs/String` | B 的语音节点 | B 的任务编排节点 |
| `/task/status` | `std_msgs/String` | B 的任务编排节点 | 三人共同验收 |

### 0.5 修改冲突处理规则

1. 任何人发现公共 Topic、关节名、目录或参数需要修改时，先在三人沟通记录中说明原因。
2. 不得为了让自己的部分运行而偷偷改动别人负责的文件。
3. 接口修改必须同时记录：
   - 修改前名称；
   - 修改后名称；
   - 修改原因；
   - 受影响文件；
   - 三人重新验证结果。
4. 最终集成阶段如果必须修改跨负责人文件，由原文件负责人执行修改。

### 0.6 公共接口冻结时间

三人完成第 1 章共同检查后，立即冻结以下内容：

```text
Topic 名称
link 名称
joint 名称
目录结构
最终任务动作顺序
```

冻结后：

- A 可以调整模型尺寸、惯量、控制器 PID 和场景位置，但不能擅自改名。
- B 可以调整识别阈值、动作时长和任务实现，但不能擅自改公共接口。
- C 可以调整 Isaac Gym 参数，但不能擅自改标准 URDF 的 link/joint 名称。

若必须解冻接口，三人应停止并行修改，先完成一次接口变更记录和同步。

## 1. 三个人都必须完成的必要步骤

本章不是“任选一个人完成”。三个人都必须理解并执行。默认三人分别使用不同 Linux 机器，因此每个人都必须在自己的机器上完成目录准备、环境检查、代码同步和记录。若三人偶尔使用同一台机器，安装动作只执行一次，但仍须分别记录负责人和验收结果。

### 1.1 阅读并理解基准文档

每个人先完整阅读：

```text
docs/具身机器人课程结课作业执行手册.md
docs/三人并行分工执行手册.md
PROJECT_STRUCTURE.md
```

每个人必须能够回答：

1. 具身机器人与传统机器人有什么区别？
2. ROS、Gazebo 和 Isaac Gym 分别负责什么？
3. 为什么 Gazebo 与 Isaac Gym 不需要同时运行？
4. 为什么不能修改已有 Conda 和 CUDA 环境？
5. 自己负责哪些文件，禁止修改哪些文件？
6. 自己要向另外两个人交付什么？

### 1.2 确认公共目录结构

每个人都必须将完整项目复制到自己的 Ubuntu 机器。即使只负责部分文件，也不能只复制自己的文件：

```text
~/robot_course/
├── catkin_ws/src/embodied_robot_demo/  # A、B 按文件所有权协作的完整 ROS 包
├── isaacgym_demo/                      # C 负责的 Isaac Gym 演示源码
├── docs/
│   ├── 具身机器人课程结课作业执行手册.md
│   ├── 三人并行分工执行手册.md
│   ├── team_inputs/
│   │   ├── person_A/
│   │   ├── person_B/
│   │   └── person_C/
│   └── final/
├── collaboration/
│   ├── handoff_template/
│   └── handoffs/
│       ├── person_A/
│       ├── person_B/
│       ├── person_C/
│       └── final/
├── evidence/
│   ├── team_common/
│   ├── person_A/
│   ├── person_B/
│   ├── person_C/
│   └── final_integration/
├── isaacgym/                           # 本地安装目录，不跨机器同步
└── envs/                               # 本地环境目录，不跨机器同步
```

检查：

```bash
cd ~/robot_course
find catkin_ws/src/embodied_robot_demo -maxdepth 3 -type f | sort
find isaacgym_demo -maxdepth 3 -type f | sort
find docs collaboration -maxdepth 3 -type f | sort
```

三人应看到相同代码版本。若文件数量或内容不同，先按照第 1.8 或第 1.9 节统一代码，不要开始各自实验。

目录设计原则：

- 不能把 A 和 B 的源码拆成两个 ROS 包。`embodied_robot_demo` 必须保持完整，否则 `roslaunch`、`rospack` 和包依赖会失效。
- A、B 通过文件所有权避免冲突，而不是通过拆散运行源码避免冲突。
- `docs/team_inputs/person_X/` 用于提交给 C 的章节、图表和说明。
- `collaboration/handoffs/person_X/` 用于跨机器传递版本、日志和交付记录。
- `evidence/person_X/` 用于保存各自机器产生的实验依据。
- `docs/final/` 和 `evidence/final_integration/` 只能保存最终集成版本的材料。

### 1.3 共同遵守环境保护规则

三个人都禁止执行：

```bash

sudo apt full-upgrade
sudo apt install nvidia-driver-*
sudo apt install cuda
sudo apt install nvidia-cuda-toolkit
pip install <任何包>        # 未确认当前环境时禁止
```

每次执行 Python 或安装依赖前必须检查：

```bash

```

```text
/usr/bin/python3
```

Isaac Gym 终端应使用：

```text
~/robot_course/envs/isaacgym/bin/python
```

语音节点终端应使用：

```text
~/robot_course/envs/voice/bin/python
```

### 1.5 三台机器分别检查角色运行环境与代码包

A、B 必须在各自机器执行完整 ROS/Gazebo 检查：

```bash
conda deactivate 2>/dev/null || true
source /opt/ros/noetic/setup.bash
source ~/robot_course/catkin_ws/devel/setup.bash

rosversion -d
gazebo --version
rospack find embodied_robot_demo
```

预期：

```text
noetic
Gazebo 11.x
~/robot_course/catkin_ws/src/embodied_robot_demo
```

若 `devel/setup.bash` 不存在：

- A 在自己的机器首次编译，用于开发 Gazebo 基线。
- B 在自己的机器独立编译，用于运行控制和多模态集成。
- C 不需要运行 Gazebo 时，可以不编译完整 catkin 工作空间，但必须能够执行 `check_urdf`。若 C 的机器没有该命令，可安装最小工具：

  ```bash
  sudo apt update
  sudo apt install liburdfdom-tools
  check_urdf ~/robot_course/isaacgym_demo/assets/embodied_robot.urdf
  ```

不同机器上的 `build/`、`devel/` 和日志目录不得互相复制。它们包含机器相关路径，复制后容易产生错误。每台机器必须从源码独立构建。

- 如果没有任何一台机器能够同时运行 Gazebo、多模态和 Isaac Gym，应选择两台演示机，但最终代码版本和材料仍必须统一。

### 1.8 代码版本管理：优先使用 Git

三人使用不同机器时，不能通过反复发送同名文件夹管理版本。推荐建立一个三人都能访问的 Git 仓库。

由集成负责人创建远程仓库并向三人提供仓库地址。三人首次同步时，在各自机器执行：

```bash
cd ~
git clone <集成负责人提供的仓库地址> robot_course
cd ~/robot_course
git rev-parse HEAD
git status --short
```

如果项目目录已经存在，不要直接在旧目录上覆盖解压。先确认旧目录中的实验记录和未提交修改已经保存，再按照团队确定的方式同步。

建议分支：

```text
main                 最终稳定版本，只合并已验收成果
person-a-gazebo      A 的工作分支
person-b-multimodal  B 的工作分支
person-c-isaacgym    C 的工作分支
```

每个人开始工作前：

```bash
cd ~/robot_course
git status --short
git branch --show-current
git rev-parse HEAD
```

记录输出中的提交哈希。实验记录中的“使用的代码版本”必须填写该哈希。

每个人只提交自己负责的源码和文档素材。禁止提交：

```text
catkin_ws/build/
catkin_ws/devel/
__pycache__/
大型视频
Isaac Gym 安装目录
Conda 环境目录
```

交付前检查：

```bash
git status --short
git diff --stat
git rev-parse HEAD
```

分支合并规则：

1. A、B、C 先在自己的分支完成本地验证。
2. 提交交付记录、测试结果和提交哈希。
3. 最终集成负责人将负责人分支合并到 `main`。
4. 每次合并后在集成机重新构建和测试。
5. 合并冲突由发生冲突文件的原负责人处理。

### 1.9 无法使用 Git 时的压缩包方案

如果三台机器无法访问同一个 Git 仓库，使用带版本号的源码压缩包。不得传递同名覆盖包。

命名格式：

```text
robot_course_v主版本.次版本.修订版本_负责人_日期.zip
```

示例：

```text
robot_course_v1.1.0_A_20260607.zip
robot_course_v1.2.0_B_20260608.zip
robot_course_v1.3.0_C_20260608.zip
```

压缩包必须只包含源码、手册和小型日志，不包含构建目录、Conda 环境、Isaac Gym 安装目录和大型视频。

创建校验值：

```bash
sha256sum robot_course_v1.1.0_A_20260607.zip \
  > robot_course_v1.1.0_A_20260607.zip.sha256
```

接收者验证：

```bash
sha256sum -c robot_course_v1.1.0_A_20260607.zip.sha256
```

压缩包中必须包含 `VERSION.txt`：

```text
负责人：
版本号：
日期：
基础版本：
修改文件：
公共接口是否变化：
执行过的测试：
已知问题：
```

### 1.10 标准交付包

无论使用 Git 还是压缩包，每次跨机器交付都必须附带：

```text
collaboration/handoffs/person_X/v版本号_日期/
├── VERSION.txt
├── CHANGELOG.md
├── environment.txt
├── commands.txt
├── test-results.md
├── logs/
├── screenshots/
└── videos/
```

创建交付包时，从模板复制：

```bash
cd ~/robot_course
cp -r collaboration/handoff_template \
  collaboration/handoffs/person_X/v版本号_日期
```

各文件要求：

- `VERSION.txt`：代码版本、负责人和基础版本。
- `CHANGELOG.md`：改了什么，为什么修改。
- `environment.txt`：实际运行机器环境。
- `commands.txt`：真实执行过的命令。
- `test-results.md`：成功、失败和已知问题。
- `logs/`：关键文本日志。
- `screenshots/`：关键截图。
- `videos/`：视频可以单独传输，但必须在交付包中记录文件名和校验值。

接收者必须在自己的机器重新执行核心测试。发送者机器成功不代表接收者机器必然成功。

### 1.11 指定最终集成机

开始并行工作前，三人必须指定最终集成机和集成负责人。

推荐选择顺序：

1. NVIDIA GPU 能运行 Isaac Gym。
2. 能安装并运行 ROS Noetic 与 Gazebo 11。
3. 麦克风可用。
4. 磁盘空间充足。
5. 答辩当天能够使用。
